# 04 Storage and Filesystem

## この章のテーマ

コンピューターは、CPUやメモリだけでは動き続けることができません。

作成したファイル、インストールしたソフトウェア、設定ファイル、ログなどは、電源を切っても残る場所に保存されます。

この章では、Linux上で次のことを観察します。

- ストレージとは何か
- ファイルシステムとは何か
- ディレクトリ構造とは何か
- マウントとは何か
- 容量や使用状況をどう確認するか
- inodeとは何か

---

## 1. ストレージとは

ストレージとは、データを保存するための装置です。

代表的なストレージには、次のようなものがあります。

- HDD
- SSD
- USBメモリ
- SDカード
- 仮想ディスク

Linuxでは、これらの装置も「ファイル」のように扱われます。

---

## 2. ストレージデバイスを確認する

まず、Linuxが認識しているストレージを確認します。

```bash
lsblk
```

`lsblk` は、`block device` を一覧表示するコマンドです。

`block device` とは、データを一定の大きさのまとまりで読み書きする装置のことです。

例：

```text
NAME   MAJ:MIN RM  SIZE RO TYPE MOUNTPOINTS
sda      8:0    0   64G  0 disk
├─sda1   8:1    0    1G  0 part /boot
└─sda2   8:2    0   63G  0 part /
```

ここでは、次のように読み取れます。

- `sda` はディスク全体
- `sda1` や `sda2` はパーティション
- `/` や `/boot` はマウント先

---

## 3. パーティションとは

パーティションとは、1つのストレージを区切った領域です。

たとえば、1つのSSDを次のように分けることができます。

- OSを入れる領域
- 起動用の領域
- データ保存用の領域

Linuxでは、パーティションごとにファイルシステムを作成して利用します。

確認してみましょう。

```bash
sudo fdisk -l
```

または、

```bash
lsblk -f
```

を使うと、ファイルシステムの種類も確認できます。

---

## 4. LVM（Logical Volume Manager）とは

LVMは、物理ディスクやパーティションを柔軟に管理する仕組みです。

通常のパーティションは、作成後にサイズ変更が難しいことがあります。

LVMを使うと、次のような操作がしやすくなります。

- 複数ディスクを1つのボリュームとしてまとめる
- 論理ボリュームのサイズを後から拡張・縮小する
- スナップショットを作成して保守作業に備える

LVMでは主に次の3層で考えます。

- PV（Physical Volume）：物理ディスクやパーティション
- VG（Volume Group）：PVを束ねたグループ
- LV（Logical Volume）：実際にファイルシステムを作る論理領域

関係は次のようになります。

```text
物理ディスク/パーティション(PV) -> ボリュームグループ(VG) -> 論理ボリューム(LV)
```

LVMの状態確認によく使うコマンドは次の通りです。

```bash
sudo pvs
sudo vgs
sudo lvs
```

`lsblk` でも、LVMが使われている場合は `TYPE` に `lvm` と表示されます。

---

## 5. ファイルシステムとは

ファイルシステムとは、ストレージ上にファイルやディレクトリを保存・管理するための仕組みです。

代表的なファイルシステムには、次のようなものがあります。

| ファイルシステム | 主な用途 |
| --- | --- |
| ext4 | Linuxでよく使われる |
| xfs | RHEL系、AlmaLinuxなどでよく使われる |
| btrfs | スナップショットなど高度な機能を持つ |
| vfat | USBメモリなどで使われることがある |
| ntfs | Windowsでよく使われる |

現在の環境で使われているファイルシステムを確認します。

```bash
df -T
```

---

## 6. マウントとは

マウントとは、ストレージやパーティションをLinuxのディレクトリ構造の中に接続することです。

Linuxでは、Windowsのように `C:` や `D:` というドライブ名では管理しません。

Linuxでは、すべてのファイルやディレクトリは `/` を頂点とする1つのツリー構造の中に配置されます。

例：

```text
/
├── boot
├── home
├── var
├── etc
└── tmp
```

別のストレージであっても、どこかのディレクトリに接続されます。

マウント状況を確認します。

```bash
mount
```

情報が多い場合は、`grep` と組み合わせます。

```bash
mount | grep "^/dev"
```

または、

```bash
findmnt
```

を使うと見やすく確認できます。

---

## 7. ディスク容量を確認する

ディスクの使用量を確認するには、`df` コマンドを使います。

```bash
df -h
```

`-h` は human readable の意味です。

人間が読みやすい単位で表示されます。

例：

```text
Filesystem      Size  Used Avail Use% Mounted on
/dev/sda2        63G   12G   51G  20% /
```

ここでは、次のように読み取れます。

- 全体容量：63G
- 使用済み：12G
- 空き容量：51G
- 使用率：20%
- マウント先：/

---

## 8. ディレクトリごとの使用量を確認する

`df` はファイルシステム全体の使用量を確認するコマンドです。

一方で、ディレクトリごとの容量を確認したい場合は `du` を使います。

```bash
du -sh /var
```

意味：

- `du`：disk usage
- `-s`：summary
- `-h`：human readable

複数のディレクトリを比較することもできます。

```bash
sudo du -sh /var/* 2>/dev/null
```

`/var` にはログやキャッシュなど、変化しやすいデータが保存されます。

---

## 9. inodeとは

Linuxのファイルシステムでは、ファイル名とは別に、ファイルの実体を管理する情報があります。

この管理情報を inode と呼びます。

inodeには、たとえば次のような情報が含まれます。

- 所有者
- 権限
- ファイルサイズ
- 更新時刻
- データの保存場所

inode番号を確認してみましょう。

```bash
ls -i
```

例：

```text
123456 README.md
```

この `123456` のような番号が inode番号です。

---

## 10. ファイル名とinodeの関係

ファイル名は、人間が扱いやすい名前です。

しかし、Linuxのファイルシステム内部では、inodeを使ってファイルを管理しています。

つまり、ファイル名は inode を指しているラベルのようなものです。

確認してみましょう。

```bash
touch sample.txt
ls -li sample.txt
```

次に、ハードリンクを作成します。

```bash
ln sample.txt sample_link.txt
ls -li sample.txt sample_link.txt
```

2つのファイル名が同じinode番号を指していることを確認できます。

---

## 11. シンボリックリンクとの違い

シンボリックリンクは、別のファイルへの参照です。

```bash
ln -s sample.txt sample_symlink.txt
ls -li sample.txt sample_symlink.txt
```

シンボリックリンクは、元のファイルとは別のinodeを持ちます。

確認してみましょう。

```bash
ls -l sample_symlink.txt
```

例：

```text
sample_symlink.txt -> sample.txt
```

---

## 12. /etc/fstab とは

`/etc/fstab` は、起動時にどのファイルシステムをどこにマウントするかを設定するファイルです。

内容を確認します。

```bash
cat /etc/fstab
```

または、

```bash
less /etc/fstab
```

`/etc/fstab` は重要な設定ファイルです。

誤って編集すると、OSが正常に起動しなくなる可能性があります。

この章では、編集はせず、確認だけにします。

## 14. 考察問題

次の問いに、自分の言葉で答えてください。

### Q1. ストレージとメモリの違いは何ですか？

ヒント：

- 電源を切ったときにデータが残るか
- CPUが直接作業に使う場所か
- ファイルを保存する場所か

---

### Q2. ファイルシステムは何のためにありますか？

ヒント：

- ファイル名
- ディレクトリ
- 所有者
- 権限
- 保存場所

---

### Q3. Linuxでは、なぜ `C:` や `D:` ではなく `/` から始まる構造で管理するのでしょうか？

ヒント：

- すべてを1つのディレクトリ構造の中で扱う
- 別のストレージもどこかのディレクトリに接続する
- マウントという考え方

---

### Q4. `df` と `du` の違いは何ですか？

ヒント：

- ファイルシステム全体を見る
- ディレクトリやファイル単位で見る

---

### Q5. inodeとは何ですか？

ヒント：

- ファイル名そのものではない
- ファイルの管理情報
- ハードリンクと関係がある

## 15. この章のまとめ

この章では、Linuxにおけるストレージとファイルシステムについて学びました。

重要なポイントは次の通りです。

- ストレージはデータを保存する装置である
- パーティションはストレージを区切った領域である
- LVMはストレージを柔軟に束ねて管理する仕組みである
- ファイルシステムはファイルやディレクトリを管理する仕組みである
- Linuxではすべてのファイルが `/` から始まるツリー構造の中に配置される
- マウントによって、ストレージをディレクトリ構造に接続する
- `df` でファイルシステム全体の容量を確認できる
- `du` でディレクトリごとの容量を確認できる
- inodeはファイルの管理情報である

ストレージやファイルシステムは、ログ管理、ユーザー管理、権限管理、セキュリティとも深く関係します。

この章は、今後のLinux学習の土台になります。
